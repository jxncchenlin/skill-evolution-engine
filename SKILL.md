# Skill Evolution Engine — Skill自我进化引擎

## 核心理念

人类通过反思改进自己，但速度受限于注意力和记忆。AI可以做得更好——

- **每次行动都是数据**：不只是错误，成功的模式也值得记录
- **第一性原理诊断**：不只看"出了什么bug"，而问"这个Skill的根本假设是什么？什么被违反了？"
- **跨Skill扩散**：一个Skill的改进启发另一个Skill，形成指数级进化
- **自动闭环**：感知→分析→改进→验证→扩散，无需人工驱动

---

## 五层架构 + 实时捕获层

```
实时捕获层（贯穿每一次对话）
         ↓
感知层 → 分析层 → 改进层 → 验证层 → 扩散层
  ↑                                  │
  └──────────────────────────────────┘
              持续循环
```

---

## Layer 0: 实时捕获层（贯穿每次对话）

**设计动机**：v1.0依赖工作日志的周期性扫描来收集信号，存在两个问题：(1) 信号延迟——问题发生到被发现可能隔了几天；(2) 依赖记忆——如果工作日志写得不详细，信号就会丢失。

AutoSkill的"主动式经验捕获"证明了一个关键洞察：**经验提取的最佳时机是任务完成的当下**。

### 0.1 触发条件

当满足以下任一条件时，引擎应在对话结束前**主动触发**经验捕获：

| 触发场景 | 示例 | 捕获动作 |
|---------|------|---------|
| **纠正成功** | AI做错了，用户纠正后重做成功 | 记录"踩坑经验"：什么错了、怎么改对的 |
| **用户满意** | "完美" "太好了" "就是这样" | 记录"成功模式"：做对了什么、什么参数有效 |
| **用户要求改进** | "下次不要这样" "能不能改成XX" | 记录"行为修正"：旧行为、新期望 |
| **用户手动触发** | "记住这个" "记下来" | 原样记录用户指定的内容 |
| **任务失败** | Skill报错、产出明显不符合预期 | 记录"失败信号"：错误描述、上下文 |

### 0.2 捕获方式（非侵入式）

**关键原则：不打断用户心流。** 捕获在对话自然结束时进行，不在任务执行中间插入。

执行方式：
1. 在任务完成的回复末尾，**附注一句话**：`[进化引擎：检测到本次任务有可提取的经验，已记录到信号库]`
2. 如果信号重要性较高（如纠正成功、任务失败），可以简短说明：`[进化引擎：已记录"XX模式下应优先使用YY"的经验]`
3. 用户如果不想记录，可以说"别记录了"，引擎会跳过本次捕获

### 0.3 捕获内容格式

```markdown
### [SIG-YYYYMMDD-XXX] [Skill名称] [信号类型] (实时捕获)
- **时间**：[ISO时间]
- **信号**：[具体描述]
- **任务上下文**：[用户在做什么任务]
- **用户原话**：[如有]
- **来源**：实时捕获
- **即时评估**：是否匹配已知FP模式？[是/否]
```

### 0.5 话题指纹与切换检测

**设计动机**：借鉴AutoSkill的话题指纹机制。在一个长对话中，用户可能连续使用同一个Skill做多次调整（比如连续优化同一段代码），此时不需要每次都重新检索经验库。但如果话题突然切换（从编程切换到写作），则需要重载经验库。

#### 话题指纹生成

每次对话开始时，从用户的输入中提取核心关键词，生成**话题指纹**：

```
指纹格式：[领域标签] + [核心动词] + [关键名词]
示例：[编程] 写 Python 函数 斐波那契
示例：[写作] 写 公众号 文章 选题
示例：[设计] 画 Logo 品牌 配色
```

#### 切换检测规则

| 判断 | 条件 | 动作 |
|------|------|------|
| **同一话题** | 新指纹与当前指纹的领域标签和核心动词相同 | 不重载经验库，跳过实时捕获（除非用户手动触发或出现纠正/失败） |
| **话题切换** | 领域标签不同，或核心动词不同 | 触发经验库重载，检查是否有该话题的Skill经验 |
| **同一Skill不同任务** | 领域标签相同，但关键名词不同 | 检查是否有该Skill的通用经验（Skill Experience层），不加载具体任务经验 |

#### 价值

- 避免同一话题内的重复检索，节省token
- 话题切换时自动加载对应经验，不会遗漏
- 通过"领域标签"实现跨Skill的同类经验提醒（比如所有编程类Skill共享通用编程偏好）

### 0.6 经验主动提醒

**设计动机**：经验存下来不是目的，**在需要的时候被调用**才是目的。借鉴AutoSkill的"跨技能经验层"，当检测到Skill即将执行时，自动检查历史经验库并主动提醒。

#### 触发时机

当以下任一条件满足时，在Skill执行前输出经验提醒：

1. **Skill首次加载**：检测到Skill被触发（通过触发词或use_skill调用），加载该Skill的Skill Experience层
2. **话题切换**：话题指纹检测到切换，重载目标话题的Skill Experience
3. **效果追踪触发**：检测到有pending状态的效果追踪项关联当前Skill

#### 提醒格式

```
[经验提醒] 基于 X 次历史使用记录：
• 踩坑：[描述一个已知的坑]
  → 建议：[最佳实践]
• 偏好：[描述用户偏好]
  → 已应用
```

**规则**：
- 只在有**非空经验**时提醒，无经验时静默
- 每次最多提醒3条，避免信息过载
- 优先级：踩坑记录 > 最佳实践 > 用户偏好
- 提醒内容应为**可执行的简短指令**，不是长篇大论

#### 提醒与改进信号的关系

经验提醒的**最终目的是触发改进**：
- 如果经验提醒后，用户仍然遇到同类问题 → 说明当前经验不够，生成新的改进信号
- 如果经验提醒被用户确认有效 → 该经验的置信度+1
- 如果用户纠正了经验内容 → 记录为新的"行为修正"信号，更新经验库

### 0.7 与感知层的关系

- 实时捕获层是**轻量级、即时性**的信号收集，关注单次任务中的经验
- 感知层（Layer 1）是**深度、周期性**的信号扫描，关注跨任务的系统性模式
- 两者互补：实时捕获保证不遗漏，感知层保证有深度

## Layer 1: 感知层（收集信号）

感知层从4个来源收集改进信号，存入 `memory/signals.md`。

### 1.1 内部信号（每次使用后自动收集）

每次Skill执行后，评估以下维度并记录：

| 信号类型 | 触发条件 | 记录内容 |
|---------|---------|---------|
| **失败** | Skill执行报错、产出不符合预期 | 错误描述、上下文、复现路径 |
| **摩擦** | 用户需要多次纠正、重试、手动调整 | 摩擦点、用户原话、期望vs实际 |
| **成功** | 一次成功完成、用户满意 | 做对了什么、用户原话 |
| **缺失** | 用户要求Skill不具备的能力 | 用户想要的、当前行为的差距 |
| **效率** | Skill执行耗时过长、token浪费过多 | 耗时、token数、可优化点 |

#### 经验分层存储

借鉴AutoSkill的知识分层思想，将收集到的经验分为两层存储：

**General Knowledge（通用知识层）**—— 与特定Skill无关的用户偏好和工作习惯：

```markdown
## 通用知识

### GK-001 通用偏好
- **领域**：通用
- **内容**：用户偏好简洁直接的回答，不喜欢冗长铺垫
- **来源**：多次对话观察
- **适用范围**：所有Skill

### GK-002 工作风格
- **领域**：通用
- **内容**：用户习惯先看方案再决定，不喜欢被自动执行高风险操作
- **来源**：2026-03-22 one-person-agency使用反馈
- **适用范围**：所有涉及自动执行的Skill
```

**Skill Experience（技能经验层）**—— 与特定Skill绑定的踩坑记录和最佳实践：

```markdown
## 技能经验

### SE-001 idol-roundtable 偶像加载经验
- **Skill**：idol-roundtable
- **经验类型**：踩坑记录
- **内容**：全量加载9个偶像文件浪费token，应按议题匹配
- **最佳参数**：每次只加载3-5个相关偶像
- **来源**：2026-03-22 首次进化循环
- **关联FP模式**：FP-002

### SE-002 multi-agent-research 子Agent分工经验
- **Skill**：multi-agent-research
- **经验类型**：踩坑记录
- **内容**：code-explorer子Agent没有web_search能力，数据收集必须在主Agent层完成
- **最佳参数**：主Agent负责数据收集→子Agent只做文件分析
- **来源**：2026-03-22 首次进化循环
- **关联FP模式**：FP-006
```

**两层协作规则**：
- 每次Skill加载时，先加载General Knowledge（通用偏好），再加载该Skill的Skill Experience
- 通用知识通常不触发改进（已是用户稳定偏好），技能经验是改进信号的主要来源
- 当某个通用知识在多个Skill中反复被验证有效时，可考虑将其写入相关Skill的SKILL.md作为默认行为

### 1.2 跨Skill信号（定期扫描）

读取其他Skill的使用记录，识别模式：

- 多个Skill出现同类问题 → 系统性问题，需要框架级修复
- 一个Skill的改进模式可迁移到另一个Skill → 扩散候选
- Skill之间的交互摩擦（如A的输出格式B无法解析）→ 接口标准化

### 1.3 用户反馈信号（实时收集）

监控用户对话中的改进意图：

| 表达模式 | 信号类型 | 示例 |
|---------|---------|------|
| 纠正 | "不对，应该..." "我不是这个意思" | 行为修正 |
| 增强 | "能不能也..." "加一个..." | 功能扩展 |
| 简化 | "太复杂了" "能不能简单点" | 体验优化 |
| 比较 | "XX做得更好" "像XX那样" | 外部对标 |
| 抱怨 | "每次都要..." "又是这个问题" | 反复出现的痛点 |

### 1.4 外部灵感信号（定期扫描）

从外部生态发现新思路，存入 `memory/external-inspiration.md`：

**触发方式**：手动触发（"扫描外部灵感"）或定期自动化（每周一次）

**扫描来源**：

| 来源 | 扫描内容 | 价值提取 |
|------|---------|---------|
| GitHub Trending (Skills/Prompts) | 爆火的Agent Skill/Prompt工程仓库 | 核心设计思路、第一性原理 |
| Twitter/X #AIAgents | 社区讨论的热门Agent模式 | 实战中的痛点和解决方案 |
| Claude Code Community | WorkBuddy用户社区 | 其他用户的最佳实践 |
| 论文 (arXiv AI Agent) | Agent自我改进的最新研究 | 学术前沿的方法论 |

**扫描产出**：每条灵感记录格式：
```markdown
## [INS-YYYYMMDD-XXX] 来源: GitHub/Twitter/社区/论文

**标题**：[一句话描述]
**来源URL**：[链接]
**核心思路**：[2-3句话]
**第一性原理**：[它解决的根本问题是什么？]
**可迁移的模式**：[哪些思路可以借鉴到我们的Skill中？]
**适用Skill**：[product-mastery / one-person-agency / 全部 / ...]
**风险评估**：low / medium / high
```

## Layer 2: 分析层（第一性原理诊断）

对感知层收集的信号进行根因分析，从第一性原理出发。

### 2.1 诊断框架

对每条信号，回答以下问题链：

```
1. 表面问题是什么？
   → [具体症状描述]

2. 这个Skill的根本假设是什么？
   → [Skill设计中隐含的、可能被违反的前提]

3. 哪个假设被违反了？
   → [被打破的假设]

4. 违反的原因是什么？（4种根因）
   → a. 假设本身错误（设计缺陷）
   → b. 假设曾正确，但环境变了（需求漂移）
   → c. 假设正确，但实现有偏差（实现bug）
   → d. 假设正确，实现正确，但用户期望不同（对齐问题）

5. 修复方向是什么？
   → [基于根因的改进方向]
```

### 2.2 优先级评分（VFM Protocol v2）

基于 proactive-agent 的 VFM 思想，扩展为面向Skill进化的评分，并引入**模式匹配快速通道**。

#### 标准VFM评分

| 维度 | 权重 | 问题 |
|------|------|------|
| 频率 | 3x | 这个问题多久出现一次？ |
| 影响面 | 3x | 影响多少个Skill？多少次使用？ |
| 自动可修复 | 2x | 能否不惊动用户、自动修改并验证？ |
| 复合杠杆 | 2x | 修一次后，是否持续受益？ |
| 扩散价值 | 2x | 这个改进的模式能否迁移到其他Skill？ |

**总分 ≥ 50** → 值得改进
**总分 ≥ 70** → 高优先级，立即执行

#### 模式匹配快速通道（Pattern Express Lane）

**问题**：首次运行中，两个已验证模式匹配的改进（VFM=31和30）被阈值过滤，但它们根因明确、修复成本极低、风险可控。

**规则**：当信号同时满足以下条件时，跳过VFM阈值，直接进入改进层：

1. **模式已验证**：信号匹配到 `first-principles-patterns.md` 中已有FP编号的模式
2. **根因一致**：当前信号的根因类型与模式定义的根因类型一致
3. **风险可控**：风险评估为Low或Medium
4. **无特殊情况**：目标Skill没有与该改进矛盾的自定义约束

**快速通道的VFM记录**：仍然计算并记录VFM分数，但标注"快速通道"，积累数据用于评估模式库的准确性。

#### 阈值动态调整

进化引擎自身也通过数据驱动改进阈值：
- 如果连续5次快速通道改进全部成功（0回滚），提高标准VFM阈值+5
- 如果标准通道改进回滚率>20%，降低阈值-5
- 阈值范围锁定在 30-60 之间，避免极端值

### 2.3 产出

分析结果写入 `memory/analysis-log.md`，格式：
```markdown
## [ANL-YYYYMMDD-XXX] [Skill名称] 信号类型

**信号来源**：[SIG/INS/USR编号]
**表面问题**：[症状]
**根本假设**：[被违反的假设]
**根因类型**：设计缺陷 / 需求漂移 / 实现bug / 对齐问题
**修复方向**：[改进方向]
**VFM评分**：频率(F)×3 + 影响面(I)×3 + 自动可修复(A)×2 + 复合杠杆(L)×2 + 扩散价值(D)×2 = [总分]
**风险评估**：low / medium / high
**快速通道**：是/否（匹配FP-XXX模式，置信度=N）
```

### 2.4 观察项自动升级

**设计动机**：v1.0首次运行中，"偶像圆桌输出不稳定"（VFM=21）被标记为观察项后就没有后续了。观察项不应永远沉睡——随着数据积累，它可能升级为可执行的改进。

**升级条件**（满足任一即升级）：

| 条件 | 说明 | 升级动作 |
|------|------|---------|
| **同类信号再次出现** | 同一Skill、同一模式的问题被实时捕获层再次记录 | 重新计算VFM（频率分+2），进入分析层 |
| **用户明确要求** | 用户在任何对话中提到该观察项的问题 | 直接进入改进层 |
| **模式置信度达到高置信** | 关联的FP模式置信度≥5 | 自动升级，按快速通道处理 |
| **周期性审查** | 每周进化循环中检查观察项是否超过30天未处理 | 标记为"过期"或升级为待改进 |

**升级流程**：
1. 从观察项中提取原始信号信息
2. 检查是否有新的同类信号（实时捕获层积累的）
3. 合并新旧信号，重新评估VFM
4. 如果VFM达标或匹配快速通道条件，执行改进
5. 如果仍不达标，重置30天观察期并说明原因

## Layer 3: 改进层（执行修改）

### 3.1 风险分级

| 风险级别 | 定义 | 执行方式 | 示例 |
|---------|------|---------|------|
| **Low** | 纯文本修改，不影响Skill结构和逻辑 | **自动执行**，无需确认 | 优化提示词措辞、补充示例、修正typo |
| **Medium** | 改变Skill行为但不影响核心流程 | 自动执行，事后告知用户 | 调整默认参数、增加判断条件、补充约束 |
| **High** | 改变Skill核心流程、结构或删除功能 | **提案给用户，确认后执行** | 修改工作流步骤、拆分/合并Skill、改变触发条件 |

### 3.2 自动执行（Low/Medium风险）

执行流程：
1. 读取目标Skill的SKILL.md和相关文件
2. 生成修改方案（old_str → new_str）
3. 执行修改
4. 更新版本号（patch: 1.1.0→1.1.1, minor: 1.1.0→1.2.0）
5. 进入验证层

### 3.3 用户提案（High风险）

提案格式：
```markdown
## 🔧 Skill改进提案

**目标Skill**：[名称] v[当前版本]
**风险级别**：High
**改进原因**：[第一性原理诊断结果]
**修改内容**：
1. [具体改动1]
2. [具体改动2]
**预期效果**：[改进后的行为变化]
**影响范围**：[哪些使用场景会受影响]
**回滚方案**：[如何恢复]
```

用户确认后进入改进层执行。

### 3.4 版本管理

每次修改遵循语义化版本：
- **Patch** (x.x.X)：Low风险修改（措辞优化、示例补充、typo修正）
- **Minor** (x.X.0)：Medium风险修改（行为调整、新增非核心流程）
- **Major** (X.0.0)：High风险修改（核心流程变更、结构重组）

---

## Layer 4: 验证层（确认改进有效）

### 4.1 自动验证清单

每次改进后，自动执行以下检查：

| 检查项 | 方法 | 通过标准 |
|--------|------|---------|
| **格式完整** | 读取修改后的SKILL.md | YAML frontmatter完整、章节结构完整 |
| **无断链** | 检查references/memory路径 | 所有引用的文件路径存在 |
| **逻辑一致** | 通读修改部分 | 修改后的内容与Skill其他部分不矛盾 |
| **触发词有效** | 检查description中的触发词 | 触发词不与其他Skill冲突 |
| **无回退** | 对比修改前后 | 新版本没有丢失旧版本的正确行为 |

### 4.2 回滚机制

如果验证不通过：
1. 立即回滚到修改前的版本
2. 记录失败原因到分析日志
3. 重新分析根因，调整改进方案

### 4.3 效果追踪（改进后的运行时验证）

**设计动机**：v1.0的验证层只做格式检查，无法回答最关键的问题——"改进真的生效了吗？"。需要一个机制在改进后**下一次使用目标Skill时**验证效果。

#### 4.3.1 追踪方式

每次改进完成后，在改进记录中写入一条**效果追踪指令**，格式：

```markdown
**效果追踪**：
- 目标Skill：[名称]
- 改进版本：v[X.Y.Z]
- 追踪问题：[这次改进要解决什么问题？]
- 预期行为：[改进后应该看到什么变化？]
- 状态：pending
```

#### 4.3.2 触发时机

当**实时捕获层**（Layer 0）检测到已改进的Skill被使用时：
1. 检查该Skill是否有 pending 状态的效果追踪指令
2. 如果有，在本次使用中关注追踪问题：
   - 同类问题是否再次出现？（频率维度）
   - 预期行为是否实现？（效果维度）
   - 是否引入新问题？（副作用维度）
3. 将追踪结果写入 `memory/signals.md`，标记为"效果验证信号"

#### 4.3.3 效果评估结论

| 结论 | 条件 | 后续动作 |
|------|------|---------|
| **改进有效** | 问题消除，无新问题 | 追踪标记为 ✅ confirmed，关联的FP模式置信度+1 |
| **部分有效** | 问题减轻但未消除 | 追踪标记为 ⚠️ partial，生成新的改进信号 |
| **无效** | 问题仍然存在 | 追踪标记为 ❌ failed，进入回滚评估；回滚率记入引擎自我评估 |
| **有副作用** | 旧问题解决但引入新问题 | 追踪标记为 ⚠️ side-effect，新问题作为新信号进入感知层 |

#### 4.3.4 置信度与模式生命周期

每个FP模式维护一个**置信度分数**：

| 事件 | 置信度变化 |
|------|-----------|
| 新模式创建 | 1 |
| 基于该模式的改进有效 | +1 |
| 基于该模式的改进无效 | -2 |
| 基于该模式的改进部分有效 | +0（不变） |
| 模式被扩散且扩散成功 | +1 |

**置信度 ≤ 0**：模式标记为"待复审"，下次匹配时要求更高风险评估
**置信度 ≥ 5**：模式标记为"高置信"，快速通道匹配时可降低风险评估门槛

## Layer 5: 扩散层（跨Skill模式传播）

这是加速进化的关键——一个Skill的改进，应该启发其他Skill。

### 5.1 扩散触发条件

当一个改进满足以下任一条件时，启动扩散分析：

- 改进的根因是"设计缺陷"，且该设计模式在多个Skill中存在
- 改进的VFM评分中"扩散价值"≥ 6
- 用户明确说"这个也可以用到XX上"

### 5.2 扩散分析

对每个改进，识别：

| 维度 | 问题 |
|------|------|
| **抽象模式** | 这个改进的本质是什么？（去掉Skill特有的细节） |
| **适用范围** | 哪些Skill有相同/相似的模式？ |
| **适配难度** | 直接套用 / 需要适配 / 不适用 |
| **预期收益** | 对目标Skill的VFM评分预估 |

### 5.3 扩散执行

```
改进完成 → 提取抽象模式 → 匹配目标Skill → 生成适配方案 → 执行/提案
```

**扩散记录**写入 `memory/diffusion-log.md`：
```markdown
## [DIF-YYYYMMDD-XXX] 源Skill → 目标Skill

**源改进**：[源Skill的改进内容]
**抽象模式**：[去Skill化后的改进模式]
**目标Skill**：[列表]
**适配方案**：[每个目标的具体适配]
**执行状态**：自动执行 / 待用户确认 / 已跳过（原因）
**效果**：[改进后的效果]
```

### 5.4 扩散实例

**历史案例**（实际发生过的扩散）：
- one-person-agency 加入"需求澄清阶段" → 扩散到 multi-agent-research 的"约束确认"
- product-mastery 的"按需加载框架" → 可扩散到 idol-roundtable 的"按议题加载偶像知识库"
- self-improving-agent 的"Recurring Pattern Detection" → 可扩散到所有Skill的Troubleshooting章节

## references/first-principles-patterns.md

常见的第一性原理改进模式，供分析层参考。每个模式维护**置信度**分数，驱动快速通道和风险评估。

### FP-001：输入不充分（最常见的根因）
- **假设**：用户提供的信息足够执行任务
- **违反**：用户说"做个网站"但没说给谁用、做什么
- **修复**：在执行前加入输入验证/澄清阶段
- **置信度**：2（已扩散验证2次：one-person-agency需求澄清、multi-agent-research约束确认）
- **已扩散**：one-person-agency、multi-agent-research

### FP-002：全量加载（上下文浪费）
- **假设**：相关知识全部加载有助于决策
- **违反**：无关知识干扰判断、浪费token
- **修复**：按主题/场景拆分知识库，按需加载
- **置信度**：3（已扩散验证3次：product-mastery框架拆分、idol-roundtable偶像匹配、引擎自身模式库）
- **已扩散**：product-mastery、idol-roundtable

### FP-003：输出不稳定
- **假设**：AI每次输出格式大致相同
- **违反**：每次复盘/分析格式不一样，无法积累
- **修复**：提供固定模板，强制约束输出结构
- **置信度**：1（product-mastery案例分析模板验证通过）
- **已扩散**：product-mastery

### FP-004：范围蔓延
- **假设**：用户知道边界在哪里
- **违反**：研究/分析越做越广，偏离初衷
- **修复**：启动时明确边界和排除项
- **置信度**：1（multi-agent-research约束确认验证通过）
- **已扩散**：multi-agent-research

### FP-005：缺乏反馈闭环
- **假设**：一次执行就能满足需求
- **违反**：做完发现方向不对
- **修复**：增加阶段性确认点
- **置信度**：1（概念验证，尚未独立扩散）

### FP-006：子Agent能力边界（首次运行新增）
- **假设**：子Agent（如code-explorer）具备与主Agent相同的工具能力
- **违反**：子Agent只有搜索/读取工具，没有web_search/web_fetch等
- **修复**：明确主Agent和子Agent的能力分工，不在子Agent的prompt中要求使用它不拥有的工具
- **置信度**：1（multi-agent-research v1.2.0验证通过）
- **已扩散**：multi-agent-research
- **待扩散**：当新Skill使用task工具调用子Agent时，自动提醒应用此模式

### 模式生命周期

| 置信度 | 状态 | 含义 | 快速通道门槛 |
|--------|------|------|------------|
| ≤ 0 | ⚠️ 待复审 | 多次改进无效，模式本身可能有问题 | 不适用快速通道 |
| 1-2 | 🟢 正常 | 初步验证，谨慎使用 | 需Low风险 |
| 3-4 | 🟢 已验证 | 多次验证可靠 | 允许Medium风险 |
| ≥ 5 | ⭐ 高置信 | 高度可靠 | 可降低风险评估门槛 |

## Troubleshooting

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 信号太多，分析不过来 | 感知层过于敏感 | 提高信号阈值，只记录VFM≥30的信号 |
| 自动改进引入新问题 | Medium风险判断失误 | 将该类改进升级为High，需用户确认 |
| 扩散误伤（不适用但强制适配） | 抽象模式过于宽泛 | 扩散分析时增加"反向验证"——找到不适用的反例 |
| 进化看板长期无更新 | 缺少自动化触发 | 设置每周Automation任务 |
| 外部灵感扫描结果不相关 | 扫描关键词太泛 | 收窄扫描范围，聚焦"AI Agent"和"Skill"相关 |

## Improvement Loop

每次进化循环结束后，反思引擎自身的改进：

### v1.0 → v1.1 改进记录（2026-03-22，受AutoSkill启发）

**来源**：AutoSkill（2026-03-17发布）的核心设计启示 + v1.0首次运行暴露的问题

| v1.0问题 | AutoSkill启示 | v1.1改进 |
|---------|-------------|---------|
| VFM=50阈值过滤了2个有效改进 | AutoSkill直接执行所有匹配的技能 | 引入模式匹配快速通道，已验证模式跳过阈值 |
| 信号收集依赖周期性日志扫描 | AutoSkill每轮对话实时捕获经验 | 新增Layer 0实时捕获层，非侵入式经验提取 |
| 验证层只做格式检查，无法确认改进有效 | AutoSkill有状态比对验证机制 | 新增效果追踪，改进后下次使用时运行时验证 |
| 观察项无后续处理机制 | — | 新增观察项自动升级机制（4种升级条件） |
| FP模式无生命周期管理 | — | 每个模式维护置信度分数，驱动快速通道门槛 |

### v1.1 → v1.2 改进记录（2026-03-22，深度吸收AutoSkill设计）

**来源**：AutoSkill完整文章分析，提取3个v1.1未覆盖的设计点

| v1.1缺失 | AutoSkill设计 | v1.2改进 |
|---------|-------------|---------|
| 无话题管理，长对话中重复检索 | 关键词指纹 + 话题切换检测 | 话题指纹生成与切换检测（Layer 0.5） |
| 信号库未分层，通用偏好和技能经验混杂 | General Knowledge + Skill Experience 双层存储 | 信号库重构为两层（GK通用知识 + SE技能经验） |
| 经验存了但不主动调用 | 使用Skill时自动检查并提醒历史经验 | 经验主动提醒机制（Layer 0.6） |

### 通用反思项

1. 哪一层效率最低？是感知不够全、分析不够深、还是扩散不够快？
2. 自动执行的改进中，回滚率是多少？如果>20%，收紧自动执行标准
3. 扩散的成功率是多少？如果<50%，改进抽象模式的提取能力
4. 外部灵感转化率如何？如果连续3周无有效转化，调整扫描策略
5. 快速通道的准确率如何？如果误判率>30%，提高快速通道的匹配门槛
